UPI Fraud Detection System

A production-minded fraud monitoring platform that combines streaming data, explainable risk scoring, and operations-focused alerting for UPI transaction flows.

Overview

The system was built to analyze transaction behavior in near-real time and flag suspicious patterns before losses escalate. It blends batch analytics and streaming-style decisioning with a lightweight API for downstream fraud teams.

Business Problem

Fraud teams needed a reliable way to uncover suspicious UPI transactions without overwhelming analysts with false positives. The main challenge was balancing recall, explainability, and operational speed.

Objectives

  • Reduce financial losses from fraudulent transactions.
  • Improve alert quality for review teams.
  • Create an explainable prediction workflow that analysts could trust.

Dataset

Historical transaction logs and labeled fraud outcomes were used to model behavioral risk. The dataset included velocity features, merchant attributes, location signals, and transaction metadata.

Data Cleaning

Missing values were standardized, invalid transaction records were removed, and categorical variables were normalized. Feature windows were also created to capture behavior over time rather than single-event activity.

EDA

Exploratory analysis uncovered strong patterns in transaction frequency, device shifts, amount anomalies, and merchant risk clusters. These findings informed the feature design and monitoring thresholds.

Feature Engineering

Engineered features included amount deviation, rapid transaction bursts, merchant history, unusual geolocation changes, and time-based behavior. These signals were aggregated into rolling windows for stronger temporal context.

Model Selection

Gradient boosting and tree-based ensemble methods were compared against simpler baselines. The final choice prioritized precision-recall balance and stability under class imbalance.

Training

Training used stratified splits and cross-validated tuning to reduce variance. Model experiments were tracked in MLflow to support reproducibility and compare threshold settings.

Evaluation

Evaluation focused on recall, false-positive rate, and calibration. Threshold tuning was adjusted so risky transactions would be surfaced quickly while keeping analyst workload manageable.

Deployment

The trained scoring pipeline was exposed through a FastAPI service and packaged for lightweight deployment. It supports automated retraining triggers and monitoring hooks for new fraud patterns.

Challenges

  • Highly imbalanced fraud data.
  • Need for interpretable results rather than opaque scores.
  • Operational constraints around latency and review workflows.

Results

The project delivered a structured fraud screening workflow that improved review efficiency while keeping alert quality high. It also established a scalable foundation for future anomaly detection work.

Lessons Learned

Model performance matters, but deployment clarity matters just as much. Good fraud systems depend on end-to-end operational design, clear thresholds, and strong analyst feedback loops.

Future Improvements

Future work includes streaming inference, more adaptive feature drift monitoring, and richer case-based explanations for review agents.
